# 完成前验证

## 何时使用

- 即将声明“完成”“通过”“已修复”之前
- 提交代码、创建 PR、切换任务之前
- 接收子代理结果之后
- 任何需要下结论的时刻

## 输入前提

- 已明确当前准备声明什么成立
- 已知道能证明这个声明的验证命令
- 愿意重新运行新鲜验证，而不是依赖旧结果
- 已区分“局部通过”“阶段通过”“任务完成”分别需要哪些证据

## 执行步骤

1. IDENTIFY：确定要跑哪些命令才能证明当前声明
2. RUN：运行完整验证命令
3. READ：完整读取输出，确认退出码和通过/失败数
4. VERIFY：判断输出是否真的支撑你的声明
5. CLAIM：只有这时才能给出结论

## 强化规则

### 1. Fresh evidence only

- 只接受当前变更之后重新运行得到的验证结果
- 之前跑过、别人说通过、CI 历史绿灯，都不能替代当前证据
- 一旦代码、配置、依赖或环境变化，之前的验证结果自动失效

### 2. 声明粒度要精确

- “单元测试通过”不等于“任务完成”
- “本地通过”不等于“CI 通过”
- “修复了一个报错”不等于“回归风险已关闭”
- 声明必须与证据粒度一致，不能扩大结论

### 3. 高风险变更需要二次核验

- 涉及发布、迁移、权限、安全、计费、数据删除、跨包联动时，至少补一层独立验证
- 二次核验可以是另一条命令、另一环境验证、人工走查，或对关键路径的复跑
- 复杂任务宁可慢一点，也不要用一次侥幸通过代替可靠结论

## 成功标准

- 结论有实际命令输出支撑
- 不使用“应该”“看起来”“大概”这类措辞替代证据
- 测试、构建、lint、需求对照等相关层级都已独立确认
- 子代理结果经过主线独立复核
- 声明范围与验证范围一致
- 高风险改动完成了必要的二次核验

## 相关原则

- 证据先于断言
- 复杂任务谨慎优先于求快
- 目标导向决定验证粒度
- 历史输出不算当前验证
- 验证是门禁，不是装饰

## 与其他技能的衔接

- 与 `test-driven-development` 的红绿循环天然配合
- 与 `engineering-principles` 配合，用原则层约束“声明什么、验证到什么程度”
- `code-review-and-quality` 会审查你的验证故事是否成立
- `sdd-tdd-workflow` 的每个阶段门控都依赖这里的规则
