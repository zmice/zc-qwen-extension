# 增量实现

## 何时使用

- 实现任何跨多个文件的改动
- 按计划逐项交付功能切片
- 做重构，但又必须保持系统持续可运行
- 你已经开始想“一次写完再看”的时候

不适用于边界已经极小的单函数改动。

## 输入前提

- 已有任务列表或清晰的实现目标
- 已知道本次切片的范围边界
- 愿意在每个切片后立即测试、验证、提交

## 执行步骤

1. 先选最小的完整切片，不要一次吞整段功能
2. 只改当前任务需要的范围，避免顺手扩散
3. 每个切片都完成 `Implement -> Test -> Verify -> Commit`
4. 保持系统始终可构建、可测试、可回退
5. 一个切片结束后，再进入下一个切片

## 成功标准

- 每个切片只解决一个逻辑问题
- 每次切片完成后系统都是绿色状态
- 没有把重构、功能、配置改动混成一团
- 实现遵守简单优先和外科式修改原则

## 相关原则

- 简单优先，不为未来需求提前抽象
- 只改当前任务需要的最小范围
- 遇到失败先停下来验证，不继续堆代码

## 与其他技能的衔接

- 接在 `planning-and-task-breakdown` 之后
- 与 `test-driven-development` 配合执行红绿重构
- 每个切片结束后由 `verification-before-completion` 兜底验证
