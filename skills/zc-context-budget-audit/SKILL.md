# 上下文审计

## 何时使用

- 会话表现变慢或质量下降，但原因不明时
- 最近增加了新的 skills、agents、rules 或 MCP 工具时
- 想知道上下文预算到底被哪些组件占掉时
- `ctx-health` 提示需要进一步审计时

## 输入前提

- 已有需要审计的上下文组件集合
- 目标是量化膨胀来源，而不是泛泛觉得“上下文太长”
- 接受这是一项治理动作，不是日常每次都要做

## 执行步骤

1. 盘点各类组件的 token 开销
2. 按始终需要、按需需要、很少需要进行分类
3. 识别臃肿、重复和低收益组件
4. 输出按收益排序的压缩建议

## 成功标准

- 能明确指出主要 token 消耗点
- 有可执行的缩减建议，而不是只给感受判断
- 可估算压缩前后的上下文开销差异

## 相关原则

- 量化优先于感觉
- 上下文不是越多越好，而是越准越好
- 审计的目标是提升有效信号密度

## 与其他技能的衔接

- 是 `ctx-health` 的深度版本
- 与 `context-engineering` 配合，用于长期治理
